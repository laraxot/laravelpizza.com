# Refactor: Use Filament\Resources\Pages\Concerns\HasWizard in XotBaseWizardWidget

## Motivation
The `XotBaseWizardWidget` class previously implemented its own wizard logic, duplicating functionality already provided by Laravel Filament's `HasWizard` trait. This duplicated code made maintenance difficult and risked divergence from Filament's official API.

By adopting the official `HasWizard` trait, we align with Filament's recommended pattern:
- Use `getSteps()` to define wizard steps
- Let Filament manage the `Wizard` component lifecycle
- Keep only domain‑specific overrides (next/previous actions,submit button)

## Changes Made

1. **Added trait import**
   ```php
   use Filament\Resources\Pages\Concerns\HasWizard;
   ```

2. **Removed duplicated wizard creation**
   - Deleted `makeWizard()` method that manually built the `Wizard` component
   - Simplified `getFormSchema()` to return the component generated by the trait via `$this->getWizardComponent()`

3. **Preserved existing hooks**
   - `configureWizardNextAction()`
   - `configureWizardPreviousAction()`
   - `getWizardSubmitAction()`
   remain unchanged to allow domain‑specific customization

4. **Kept abstract `getWizardSteps()`**
   - Child widgets must still provide their step definitions
   - The trait respects this method to build the wizard

5. **Documentation**
   - Added this design decision note in `docs/wiki/XotBaseWizardWidget-HasWizard-refactor.md`
   - Will be ingested into the project QMD knowledge base for future reference

## Benefits
- **Reduced maintenance**: No manual wizard component construction
- **Synchronization**: Automatically inherits any Filament improvements to `HasWizard`
- **Clarity**: Implementation now mirrors the official Filament pattern
- **Safety**: Eliminates risk of subtle bugs from divergent wizard logic
