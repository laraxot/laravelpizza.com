## 📊 Error 1: Array Shape Type Mismatch for `by_type` Return Value

### Error Details

**Location**: `ActivityLogger.php:261`
**Method**: `getStatistics()`
**Severity**: PHPStan Level 10 Error

**PHPStan Message**:
```
Method Modules\Activity\Actions\ActivityLogger::getStatistics() return type has no value type specified
for the 'by_type' key. Expected: array<string, int>, Found: array
```

### Current Code (Lines 252-289)

```php
/**
 * Get activity statistics.
 *
 * @return array{total: int, by_type: array<string, int>, today: int, this_week: int, this_month: int}
 */
public function getStatistics(?User $user = null): array
{
    $query = Activity::query();

    if ($user) {
        $query->where('causer_id', $user->getKey())
              ->where('causer_type', $user::class);
    }

    return [
        'total' => $query->count(),
        'by_type' => (function () use ($query): array {
            /** @var \Illuminate\Database\Eloquent\Builder<Activity> $clonedQuery */
            $clonedQuery = $query->clone();

            /** @var \Illuminate\Support\Collection<int, object{event: string, count: int}> $results */
            $results = $clonedQuery
                ->selectRaw('event, COUNT(*) as count')
                ->groupBy('event')
                ->get();

            // Line 274 - Error 2 occurs here (see next section)
            return $results->mapWithKeys(function (\stdClass $item) {
                return [(string) $item->event => (int) $item->count];
            })->toArray();
        })(),
        'today' => $query->clone()
            ->whereDate('created_at', now()->toDateString())
            ->count(),
        'this_week' => $query->clone()
            ->whereBetween('created_at', [now()->startOfWeek(), now()->endOfWeek()])
            ->count(),
        'this_month' => $query->clone()
            ->whereMonth('created_at', now()->month)
            ->whereYear('created_at', now()->year)
            ->count(),
    ];
}
```

### Root Cause Analysis

**Why This Error Occurs**:

1. **Inline Closure Complexity**: The `by_type` value is generated by an inline IIFE (Immediately Invoked Function Expression) closure
2. **PHPDoc Annotation Mismatch**: The PHPDoc says `@var Collection<int, object{event: string, count: int}>` but `selectRaw()` returns `Collection<int, \stdClass>`
3. **Type Inference Limitation**: PHPStan cannot fully infer the return type through the closure → `mapWithKeys()` → `toArray()` chain
4. **Array Shape Lost in Translation**: The precise array shape `array<string, int>` is lost during the transformation

**Business Context**:

The `by_type` statistics show the breakdown of activities by event type (e.g., "created" => 150, "updated" => 87, "deleted" => 23). This data is used for:
- **Compliance Dashboards**: Showing audit trail summaries for GDPR/regulatory compliance
- **Analytics Widgets**: Displaying activity distribution charts in Filament dashboards
- **Security Monitoring**: Identifying unusual activity patterns (e.g., spike in deletions)

The type safety issue means we cannot guarantee at compile-time that the data structure matches what consumers expect, potentially causing runtime errors in dashboard widgets or API responses.

## 🔧 Error 2: Parameter Type Mismatch in mapWithKeys() Closure

### Error Details

**Location**: `ActivityLogger.php:274`
**Method**: `getStatistics()` (closure inside)
**Severity**: PHPStan Level 10 Error

**PHPStan Message**:
```
Parameter #1 $callback of method Illuminate\Support\Collection::mapWithKeys() expects
Closure(object{event: string, count: int}, int): array,
Closure(\stdClass): array given.
```

### Current Code (Line 274)

```php
return $results->mapWithKeys(function (\stdClass $item) {
    return [(string) $item->event => (int) $item->count];
})->toArray();
```

### Root Cause Analysis

**Why This Error Occurs**:

1. **PHPDoc vs Runtime Type Mismatch**:
   - **PHPDoc says**: `Collection<int, object{event: string, count: int}>`
   - **Runtime reality**: `selectRaw()` always returns `\stdClass` objects
   - **Closure parameter**: Uses `\stdClass $item` (correct runtime type)
   - **PHPStan expects**: Based on PHPDoc, expects `object{event: string, count: int}`

2. **Conflict Between Documentation and Implementation**:
   - The PHPDoc annotation attempts to provide type safety by specifying the exact shape
   - However, Laravel's query builder fundamentally returns `\stdClass` instances
   - PHPStan enforces the PHPDoc contract, creating a conflict with actual runtime behavior

**Business Context**:

This method processes raw SQL aggregate results (`SELECT event, COUNT(*) as count GROUP BY event`) which are always returned as generic `stdClass` objects by Laravel. The data represents critical audit metrics:
- **Event Distribution**: How many times each type of activity occurred
- **Compliance Reporting**: Required for audit trail completeness reports
- **Security Analytics**: Detecting anomalous activity patterns

The type mismatch doesn't affect runtime execution but prevents PHPStan from providing compile-time guarantees about data structure correctness, which is especially important for audit/compliance data.

## 🔬 Proposed Solutions

### Solution 1: Extract to Dedicated Private Method (RECOMMENDED)

**Approach**: Move the statistics-by-type logic to a separate well-typed private method.

**Implementation**:

```php
/**
 * Get activity statistics.
 *
 * @return array{total: int, by_type: array<string, int>, today: int, this_week: int, this_month: int}
 */
public function getStatistics(?User $user = null): array
{
    $query = Activity::query();

    if ($user) {
        $query->where('causer_id', $user->getKey())
              ->where('causer_type', $user::class);
    }

    return [
        'total' => $query->count(),
        'by_type' => $this->getStatisticsByType($query),
        'today' => $query->clone()
            ->whereDate('created_at', now()->toDateString())
            ->count(),
        'this_week' => $query->clone()
            ->whereBetween('created_at', [now()->startOfWeek(), now()->endOfWeek()])
            ->count(),
        'this_month' => $query->clone()
            ->whereMonth('created_at', now()->month)
            ->whereYear('created_at', now()->year)
            ->count(),
    ];
}

/**
 * Get activity statistics grouped by event type.
 *
 * @param \Illuminate\Database\Eloquent\Builder<Activity> $query
 * @return array<string, int>
 */
private function getStatisticsByType(\Illuminate\Database\Eloquent\Builder $query): array
{
    /** @var \Illuminate\Support\Collection<int, \stdClass> $results */
    $results = $query->clone()
        ->selectRaw('event, COUNT(*) as count')
        ->groupBy('event')
        ->get();

    /** @var array<string, int> $statistics */
    $statistics = $results->mapWithKeys(function (\stdClass $item): array {
        return [(string) $item->event => (int) $item->count];
    })->toArray();

    return $statistics;
}
```

**Pros**:
- ✅ Clear separation of concerns (SOLID Single Responsibility Principle)
- ✅ Explicit return type `array<string, int>` for the extracted method
- ✅ PHPStan can infer types correctly without inline closure complexity
- ✅ Easier to test in isolation
- ✅ Easier to add caching later if needed
- ✅ Uses `\stdClass` correctly (matches Laravel reality)

**Cons**:
- Adds one more method to the class (minimal overhead)

**Why This is Best**:
1. **DRY + KISS**: Simple, straightforward, maintainable
2. **Type Safety**: PHPStan Level 10 compliant with explicit types
3. **Testability**: Can unit test `getStatisticsByType()` independently
4. **Business Logic Clarity**: Method name clearly expresses intent
5. **Future-Proof**: Easy to enhance (e.g., add caching, filtering)

### Solution 3: Use Data Transfer Object (DTO)

**Approach**: Create a typed DTO for statistics results.

**Implementation**:

```php
// New file: Modules/Activity/app/Data/ActivityStatisticsData.php
<?php

declare(strict_types=1);

namespace Modules\Activity\Data;

use Spatie\LaravelData\Data;

class ActivityStatisticsData extends Data
{
    public function __construct(
        public readonly int $total,
        /** @var array<string, int> */
        public readonly array $by_type,
        public readonly int $today,
        public readonly int $this_week,
        public readonly int $this_month,
    ) {
    }
}

// In ActivityLogger.php
public function getStatistics(?User $user = null): ActivityStatisticsData
{
    $query = Activity::query();

    if ($user) {
        $query->where('causer_id', $user->getKey())
              ->where('causer_type', $user::class);
    }

    return new ActivityStatisticsData(
        total: $query->count(),
        by_type: $this->getStatisticsByType($query),
        today: $query->clone()->whereDate('created_at', now()->toDateString())->count(),
        this_week: $query->clone()->whereBetween('created_at', [now()->startOfWeek(), now()->endOfWeek()])->count(),
        this_month: $query->clone()->whereMonth('created_at', now()->month)->whereYear('created_at', now()->year)->count(),
    );
}
```

**Pros**:
- ✅ Maximum type safety with Spatie Laravel Data
- ✅ Consistent with Laraxot framework patterns (uses Spatie Data everywhere)
- ✅ Provides validation, casting, and transformation capabilities
- ✅ Can be serialized to JSON cleanly for API responses
- ✅ Self-documenting structure

**Cons**:
- Adds a new file (ActivityStatisticsData.php)
- Changes return type (breaking change for existing consumers)

**Why This Could Be Best Long-Term**:
If the statistics are used across multiple layers (API responses, dashboard widgets, etc.), a DTO provides the strongest type guarantees and best developer experience.
